Spring-Aop设计原理及源码分析

Posted on 2019-05-24 In Spring

有时间读读源码，学习大佬的编码洁癖，设计规范……

Spring Aop依赖于Spring IOC容器，它是在等IOC容器初始化以后，拿到IOC容器的引用，然后获取相应的Aop相关操作的对象。

Tips:前面已经介绍过了代理模式，代理模式

从工厂的角度看Spring Aop

Spring的核心设计思想就是工厂模式，Spring Bean的创建是通过BeanFactory的createBean方法，获取Bean是通过getBean的方法，因此我们可以类比Spring Aop是不是也应用了工厂方式，然后我们可以在Spring Aop源码包从找到相应的类，ProxyFactory。

ProxyFactory

我们可以看到，Aop的代理对象也是通过ProxyFactory工厂进行创建的。
然后我们更进到getBean这个方法，发现了一个AopProxy的接口.
AopProxy

我们找到这个接口的实现类，发现核心是两个，CglibAopProxy，JdkDynamicAopProxy。

即Spring 提供了两种方式来生成代理对象: JDKProxy 和 Cglib。

jdk动态代理源码分析

具体使用哪种方式生成由 AopProxyFactory 根据 AdvisedSupport 对象的配置来决定。默认的策略是如果目标类是接口，则使用 JDK 动态代理技术，否则使用 Cglib 来生成代 理。下面我们来研究一下 Spring 如何使用 JDK 来生成代理对象，具体的生成代码放在 JdkDynamicAopProxy 这个类中。
JdkDynamicAopProxy实现了InvocationHandler，是jdk实现动态代理必须实现的接口。

是不是很熟悉，跟之前我们自己实现动态代理的手法大相径庭。
先获取目标对象的接口，然后根据类加载器和接口去创建代理对象
，生成的代理对象的方法调用都会委托到 InvocationHandler.invoke() 方法。而通过 JdkDynamicAopProxy 的签名我们可以看到这个类其实也实现了 InvocationHandler，下面我们就通过分析这个 类中实现的 invoke()方法来具体看下 Spring AOP 是如何织入切面的。

过程可以理解为：

  • 如果目标对象调用的方法是equals(),hashcode(),Advise接口的方法，直接调用并返回。
  • 否则根据Advise，获取应用到此方法上的 Interceptor 列表 。
  • Interceptor 列表 为空，直接反射调用 method.invoke(target, args)
  • 否则反射生成 MethodInvocation。
  • 进行MethodInvocation的调用。

主流程可以简述为：获取可以应用到此方法上的通知链（Interceptor Chain）,如果有,则应用通知,并执行 joinpoint; 如 果没有,则直接反射执行 joinpoint。而这里的关键是通知链是如何获取的以及它又是如何执行的，下面逐一分析下。
